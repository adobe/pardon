# Overview

Pardon is, at its core, a composable dynamic templating language powering
an http framework.  Template composition is used to both match requests and extend them,
as well as control how they are rendered.

Pardon simplifies the description and use of HTTP APIs by providing a consistent design
language combined with composition to separate concerns. One concern might be the
environment: production or test, or the local machine.
Another concern might be the authorization token. By representing these concerns
separately, they become easier to manage and maintain.

Pardon's templates are designed to be both human and machine readable, but note that
the `?` in `"{{?task}}"` means that the value is optional, and the `@` in `{{@auth-token}}`
indicates a secret which will be redacted in logs (and generally hidden by default).

A pardon collection would have the above templates in `.https` files, or "HTTP-schema" files.
An http schema file represents one (or more) ways to make a request
(prefixed by `>>>` for an outbound request),
and optionally, templates to match and process the response
(prefixed by `<<<` for an inbound response).

For example, the user login schema can specify a value placeholder in the response,
demonstrated below with `{{@token}}`. This placeholder both tells Pardon to treat
this as secret data (not for long term storage), and to to parse the value from the response.

```http
>>>
PUT https://todo.example.com/users
Content-Type: application/json

{
  "username": "{{username}}",
  "password": "{{@password}}"
}

<<<
200 OK

{
  "token": "{{@token}}"
}
```

By parsing the value, a script can reference the data directly when using `pardon()`'s internal API
to access the authorization call.

```ts
import { pardon } from "pardon";

export async function authorizeUser(username: string) {
  const password = await getPassword(username);

  const {
    inbound: {
      secrets: { token },
      response: res,
    },
  } = await pardon({ username, password })`
    PUT https://todo.example.com/users
  `();

  if (!token) {
    throw new Error(
      `failed to authorize user ${username}: (${res.status}) ${res.body}`,
    );
  }

  return token;
}
```

With the above helper, we can use `Authorization: User {{@user-token = authorizeUser(username)}}`
to automatically specify the credentials for calls that need authorization,
and Pardon will take care of generating the token.

Pardon's templating is more than text processing, it is a data processing language
 - Templates both render out and process input.
 - Templates provide a _skeleton_ of a request, but they do not _constrain_ the request to that skeleton.

To get a feel for how pardon templates work, let's exercise this todo API.

 - make a request.
 - create a user.
 - create a todo item.

### Ease of use

<Tabs>
  <TabItem label="production">
Pardon can execute a simple request, like curl, but using templates reduces the noise.
```bash title="https://example.com/ping"
% pardon https://example.com/ping
pong # from prod
```
  </TabItem>
  <TabItem label="staging">
Pinging stage is just as easy.
```bash frame="terminal" title="https://stage.example.com/ping"
% pardon https://stage.example.com/ping
pong # from stage
```
  </TabItem>
  <TabItem label="staging / http">
`pardon` can emit the http request (without making the call), just add `--http`:
```bash frame="terminal" title="https://stage.example.com/ping --http"
% pardon https://stage.example.com/ping --http
GET https://stage.example.com/ping
accept: text/plain
api-key: staging
```
  </TabItem>
  <TabItem label="env=stage">
By adding `env=stage` we can rewrite the request to use stage!

Here we see the output of pardon's "`--curl`" output.
```bash frame="terminal" title="https://example.com/ping --curl env=stage"
% pardon https://example.com/ping env=stage --curl
curl "https://stage.example.com/ping" \
  --header "accept: text/plain" \
  --header "api-key: staging"
```
  </TabItem>
</Tabs>
