## Overview

Pardon is, at its core, a composable dynamic templating language powering
an http framework.  Template composition is used to both match requests and extend them,
as well as control how they are rendered.

Pardon simplifies the description and use of HTTP APIs by providing a consistent design
language combined with composition to separate concerns. One concern might be the
environment: production or test, or the local machine.
Another concern might be the authorization token. By representing these concerns
separately, they become easier to manage and maintain.

## Tutorial

Let's consider a simple multi-user TODO REST service, hosted at `https://todo.example.com/`,
with the following REST API:

<Tabs>
  <TabItem label="/ping">
  Returns `pong`. Used as a healthcheck.
  Authorization is optional.

  ```http
  GET https://todo.example.com/ping
  Authorization: {{?@authorization}}
  ```
  </TabItem>
  <TabItem label="/users">
    Requests for managing users
    <Tabs>
      <TabItem label="POST">
      Creates a user.

      ```http
      POST https://todo.example.com/users
      Content-Type: application/json

      { "username": "{{username}}", "password": "{{@password}}" }
      ```
      </TabItem>
      <TabItem label="PUT /{{user}}/login">
      Authorizes a user, returns an authorization token.

      ```http
      PUT https://todo.example.com/users/{{user}}/login
      Content-Type: application/json

      { "password": "{{password}}" }
      ```
      </TabItem>
      <TabItem label="DELETE /{{user}}">
      Deletes a user.

      ```http
      DELETE https://todo.example.com/users/{{user}}
      Authorization: User {{@authorization}}
      ```
      </TabItem>
    </Tabs>
  </TabItem>

  <TabItem label="/users/{{user}}/todos">
    Requests for managing todos
    <Tabs>
      <TabItem label="POST ">
      Creates a new todo.

      ```http
      POST https://todo.example.com/users/{{user}}/todos
      Content-Type: application/json
      Authorization: User {{@authorization}}

      { "task": "{{task}}", "completed": "{{completed}}" }
      ```
      </TabItem>
      <TabItem label="GET">
      Gets all the todos for a user.

      ```http
      GET https://todo.example.com/users/{{user}}/todos
      Authorization: User {{@authorization}}
      ```
      </TabItem>
      <TabItem label="GET">
      Gets the state of a todo

      ```http
      PUT https://todo.example.com/users/{{user}}/todos/{{todo}}
      Authorization: User {{@authorization}}
      ```
      </TabItem>
      <TabItem label="PUT /{{todo}}">
      Updates a todo.

      ```http
      PUT https://todo.example.com/users/{{user}}/todos/{{todo}}
      Content-Type: application/json
      Authorization: User {{@authorization}}

      { "task": "{{?task}}", "completed": "{{?completed}}" }
      ```
      </TabItem>
      <TabItem label="DELETE /{{todo}}">
      Deletes a todo.

      ```http
      DELETE https://todo.example.com/users/{{user}}/todos/{{todo}}
      Authorization: User {{@authorization}}
      ```
      </TabItem>
    </Tabs>
  </TabItem>
</Tabs>

These are Pardon templates.  The little bit of non-obvious
syntax is the `?` in `"{{?task}}"`, which means that the value is optional,
and the `@` in `{{@authorization}}` which means that Pardon will treat the
value as a secret and try to redact it in logs and other contexts.

Given a Pardon collection, a request to update a todo, could omit details
```http
PUT https://todo.example.com/users/tim/todos/1000

{ "completed": true }
```
or, using the template as is, Pardon can render it with specified values
```http
user=tim todo=1000 completed=true
PUT https://todo.example.com/users/{{user}}/todos/{{todo}}

{ "completed": "{{completed}}" }
```
or, pardon can render the endpoint by name, http method, and values only.
```
user=tim todo=1000 completed=true
endpoint=todoservice/todo/update
```

All of the above will produce the same request.
```http
PUT https://todo.example.com/users/tim/todos/1000
Content-Type: application/json
Authorization: {{@authorization}}

{ "completed": true }
```

(We will come back to scripting `{{@authorization}}` later).

<Tabs>
  <TabItem label="production">
Pardon can execute a simple request, like curl, but using templates reduces the noise.
```bash title="https://example.com/ping"
% pardon https://example.com/ping
pong # from prod
```
  </TabItem>
  <TabItem label="staging">
Pinging stage is just as easy.
```bash frame="terminal" title="https://stage.example.com/ping"
% pardon https://stage.example.com/ping
pong # from stage
```
  </TabItem>
  <TabItem label="staging / http">
`pardon` can emit the http request (without making the call), just add `--http`:
```bash frame="terminal" title="https://stage.example.com/ping --http"
% pardon https://stage.example.com/ping --http
GET https://stage.example.com/ping
accept: text/plain
api-key: staging
```
  </TabItem>
  <TabItem label="env=stage">
By adding `env=stage` we can rewrite the request to use stage!

Here we see the output of pardon's "`--curl`" output.
```bash frame="terminal" title="https://example.com/ping --curl env=stage"
% pardon https://example.com/ping env=stage --curl
curl "https://stage.example.com/ping" \
  --header "accept: text/plain" \
  --header "api-key: staging"
```
  </TabItem>
</Tabs>

Let's explore this further, in a playground environment:
