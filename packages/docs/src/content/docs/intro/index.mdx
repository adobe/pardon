Pardon is a polite http framework.

Pardon is a template processor:
templates are used to describe, match, format, extend, and parse
http requests and the responses.
Templates are written in a dialect of the "HTTP format".

Templates supply the "etiquette" that is easy to forget.
"Did you remember your content-type header?".  Yeah, of course you did,
you were focused on the data.

Authorization tokens? Request signing? Switching environments?
All easily done in Pardon, and yet nothing is built-in.

## Let's dig in

To demonstrate, we'll need a context, a multi-user CRUD TODO app.

- This API manages users and todos.
- The login API exchanges password-based authentication for a bearer token.
- Logged in users can edit their todo list (create, read, update, delete, and list). 
- Todo objects have a task (text description) and a "done" flag.

Please familiarize yourself with the following 8 requests to manage users and todos.

A pardon collection would contain the above templates in `.https` files,
or "HTTP-schema" files of request and response templates.
{/* Yes, I'm quite proud of myself for that name, thanks for asking  */}

Response templates extract and redact values in a response,
for example, the user login HTTP schema can specifies a 
redacted-value placeholder in the response:

```http
>>>
PUT https://todo.example.com/users
Content-Type: application/json

{
  "username": "{{username}}",
  "password": "{{@password = getPassword(username)}}"
}

<<<
200 OK

{
  "token": "{{@token}}"
}
```

The `{{@token}}` is important for two reasons, primarily it identifies the
field as a secret which masks it in logs, but also a script can use the
extracted value when using `pardon()`'s Javascript API to access the authorization call.

Here's an example of a script that makes another request.

```ts
import { pardon } from "pardon";

export async function authorizeUser(username: string) {
  const {
    ingress: { response: res, secrets: { token } },
  } = await pardon({ username })`
    PUT https://todo.example.com/users
  `();

  if (!token) {
    throw new Error(
      `failed to authorize user ${username}: (${res.status}) ${res.body}`,
    );
  }

  return token;
}

export async function getPassword(username: string) {
  return ...password for user...;
}
```

First we created a user with name "test" using input values.

You can see pardon also supplies the necessary but forgettable `content-type` header.
It's not a guess, nor is it based on the request body: it's in the template.  

After sending this out, the server-emulator in this page "registered" the test user.

Next we created a todo: if you go back to that second request and unlock the
secrets for viewing, you will see

```http ins={2}
POST https://todo.example.com/todos
authorization: User jwt.eyJ1c2VybmFtZSI6InRlc3QifQ==
content-type: application/json

{
  "task": "learn pardon"
}
```
(while this authorization token is just `{"username":"test"}` in base64, it is being
generated from a simulated login call.)

The https (schema) for this request could look like this

```http {1-3} /[{]{2}@.*[}]{2}/
import:
  ./authorization.ts:
    - authorizeUser
>>>
POST https://todo.example.com/todos
Content-Type: application/json
Authorization: User {{@token = authorizeUser(username)}}

{
  task: "{{task}}",
  done: "{{?done}}"
}

<<<
200

{
  "id": "{{+todo}}"
}
```

At the top, we see a yaml-format configuration section.  This config section
provides "rendertime" values, along with defaults and other configuration for the request.

Here the config imports the method used in the template, `{{@token = authorizeUser(username)}}`.

Looking at the request and response, we can see that some of the variables have "hints", 
the plain `{{task}}` is mandatory input.  However, the `{{?done}}` is marked optional
by the `?`-mark.  In the response, the `+` in `{{+todo}}` marks a dataflow output,
and it's why the `todo=T1001` was added automatically for the next request.

Feel free to explore the other rest calls, listing todos,
deleting todos and/or users.
